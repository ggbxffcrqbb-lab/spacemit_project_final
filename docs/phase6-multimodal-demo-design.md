适用日期：`2026-07-01`

## 2. 最终演示的一句话定位

**在无网工业巡检场景下，Muse Pi Pro 本地同时完成视觉缺陷感知、专业知识检索、专家问答和语音播报，形成一个全离线的防腐巡检专家终端。**

这个定位必须始终围绕三个关键词：

1. `全离线`
2. `板端本地闭环`
3. `多模态联动而不是单点 Demo`

## 3. 推荐的比赛主叙事

不建议把故事讲得过宽。最终比赛建议固定为一个主场景：

**“偏远海工/油气设备巡检点，现场无外网，巡检人员面对疑似腐蚀区域，需要一边看、一边问、一边得到本地专家建议。”**

这样有三个好处：

1. 能把你们现有的防腐知识库、规则卡、SOP 卡、视觉缺陷链路串成一条完整故事线。
2. 能自然解释为什么必须离线、为什么要板端部署、为什么要语音交互。
3. 能避开“泛聊天”比较，把评委注意力拉回专业能力闭环。

## 4. 最终演示形态：双层展示

最终不要只做一个窗口，建议采用 **“评委主画面 + 操作员终端面板”** 的双层形态。

### 4.1 评委主画面

面向评委的大屏，核心目标是“看懂系统正在做什么”。

推荐结构：

1. 左侧 `65%~70%`：实时相机主画面
2. 右侧 `30%~35%`：诊断与建议面板
3. 顶部：离线状态、设备状态、模式状态
4. 底部：语音转写、系统动作、当前结论

建议沿用并升级当前 `competition_display.py` 的风格，而不是另起炉灶做浏览器页。

主画面应长期显示以下固定元素：

1. `OFFLINE` 标识
2. `Muse Pi Pro / K1` 标识
3. `LIVE CAMERA` 标识
4. 当前视觉模式，如 `Corrosion Two-Stage`
5. 当前语音模式，如 `RAG Expert`

右侧诊断面板建议固定四块：

1. `视觉结论`
   当前疑似缺陷、子类型、置信度、区域数量
2. `专家建议`
   一句短结论 + 一句处置建议
3. `知识依据`
   仅显示 1-2 条最关键引用来源
4. `板端指标`
   分割耗时、分类耗时、TTFT、总耗时

### 4.2 操作员终端 UI

面向操作员的终端面板，核心目标是“稳定控场、快速兜底、便于解释技术亮点”。

推荐使用轻量终端 UI，而不是再做一个重 Web UI。优先建议：

`Python + rich Live` 或 `Textual-lite 风格的单进程 TUI`

原因：

1. 终端 UI 能直接在板端本地终端或 SSH 下运行。
2. 调试成本远低于再做一套浏览器前端。
3. 很适合展示实时日志、状态机、指标和快捷键。

终端 UI 建议划分为 5 个区块：

1. `系统状态栏`
   显示音频、相机、RAG、模型、温启动状态
2. `实时事件流`
   显示“开始录音 -> ASR -> 视觉上下文 -> RAG -> LLM -> TTS”
3. `最近一轮对话`
   显示用户问题、系统回答、引用来源
4. `视觉摘要`
   显示最近一帧/最近一个稳定目标的检测摘要
5. `快捷键提示`
   显示录音、讲解、生成摘要、切换模式等热键

## 5. Phase 6 的最佳主链选择

### 5.1 语音主链

正式比赛主链建议：

`ASR + qwen2.5:0.5b + RAG + Matcha TTS`

不建议把 `smollm2:135m` 作为主舞台模型，原因是：

1. `2026-06-13` 的板端基准显示它更快，但回答稳定性明显弱于 `qwen2.5:0.5b`
2. 比赛评委更看重专业可靠，不是绝对最快
3. 你的作品不是聊天助手，而是防腐专家终端

### 5.2 视觉主链

正式比赛主链建议：

`vision_usb_corrosion_two_stage_rt.yaml`

即：

1. `Stage-1` 板端腐蚀区域分割
2. `Stage-2` 腐蚀子类型分类
3. 实时画面中只展示经过稳定跟踪后的结果

不建议把 `vision_usb_defect_exp.yaml` 作为最终主视觉链，原因是：

1. 它更适合 Phase 5 的实验验证
2. 曾经出现过“样例可出框、现场画面不稳定”的问题
3. 两阶段链路更符合“先发现区域，再解释类型”的评委理解路径

### 5.3 模式切换策略

最终只保留 3 个面向演示的模式，不要太多：

1. `巡检模式`
   相机常开，持续视觉检测，等待用户发问
2. `专家问答模式`
   用户围绕当前画面或防腐知识提问，系统给出带依据回答
3. `摘要播报模式`
   系统输出当前巡检结论，用于答辩收口

## 7. 建议的比赛演示流程

建议把整场演示压缩到 `4-5 分钟`，不要超过 `6 分钟`。

### 7.1 第 0 幕：开场 20 秒

操作：

1. 展示板端主画面已处于 `OFFLINE`
2. 简短说明“不依赖云端 API，当前所有能力都在 Muse Pi Pro 本地闭环运行”

目的：

让评委先记住“全离线”和“板端”。

### 7.2 第 1 幕：视觉发现 40 秒

操作：

1. 用 USB 相机对准腐蚀样件、打印图或现场演示板
2. 主画面出现腐蚀区域、类型或风险提示

推荐台词：

“现在板端先看现场，不需要联网上传图像，就能在本地识别疑似腐蚀区域并给出类别判断。”

### 7.3 第 2 幕：语音追问 60-90 秒

推荐提问：

1. “这是哪种腐蚀？”
2. “这种情况优先怎么处理？”
3. “如果在海工高腐蚀区域，巡检时还要重点检查什么？”

系统动作：

1. ASR 转写
2. 把当前视觉结果作为上下文注入
3. RAG 检索知识库
4. LLM 组织回答
5. TTS 播报
6. 主屏显示引用来源

这是整场最关键的一幕，因为它体现的是：

`看见 -> 理解 -> 检索 -> 解释 -> 播报`

### 7.4 第 3 幕：追问与联动 40-60 秒

推荐追问：

1. “为什么你判断是这个类型？”
2. “如果现在不能立刻停机，临时措施是什么？”

此处重点不是再秀视觉，而是让评委看到：

**视觉结果不是孤立标签，而是能被问答模块继续解释。**

### 7.5 第 4 幕：摘要收口 30-40 秒

系统自动或按键生成一段简短摘要：

1. 当前发现了什么
2. 风险级别如何
3. 下一步建议做什么
4. 依据来自哪两条知识

这一幕适合作为答辩收尾。

## 8. Phase 6 真正要实现的“联动”

Phase 6 不是把 `voice-console` 和 `vision-stream` 同时打开，而是要实现 **共享状态与联合解释**。

建议新增以下板端模块：

1. `app/core/session_state.py`
   存放统一的当前会话状态
2. `app/core/event_bus.py`
   负责广播语音事件、视觉事件、UI 事件
3. `app/core/multimodal_controller.py`
   负责把视觉结果注入问答上下文
4. `app/ui/terminal_dashboard.py`
   输出操作员终端 UI
5. `configs/multimodal_demo.yaml`
   统一 Phase 6 演示配置
6. `scripts/launch_phase6_demo.sh`
   一条命令起最终演示

## 9. 视觉结果如何喂给问答模块

建议把视觉输出先规整成一条简洁摘要，而不是直接把原始 JSON 塞进提示词。

推荐格式：

```text
当前画面摘要：
- 稳定目标数：2
- 目标1：点蚀倾向，置信度 0.94，区域偏小
- 目标2：缝隙腐蚀倾向，置信度 0.53，区域纵向延伸明显
- 当前建议：优先人工复核点蚀区域，记录位置与范围
```

然后只在用户问题明显依赖当前画面时注入，例如：

1. “这是什么腐蚀？”
2. “现在这个情况怎么处理？”
3. “这个地方危险吗？”

如果用户问的是纯知识问题，则走普通 `RAG + LLM` 路径，不强行加视觉上下文。

## 10. 比赛可靠性设计

最终演示一定要有 `A/B/C` 三层兜底。

### A 方案：正式版

`实时相机 + 两阶段视觉 + 语音问答 + RAG + TTS + 主画面联动`

这是主舞台。

### B 方案：稳妥版

`静态缺陷样例图 + 两阶段视觉单图结果 + 实时语音问答 + RAG + TTS`

如果现场光照、反光、对焦不理想，立即切到这套。

### C 方案：保底版

`视觉结果预先缓存 + 语音问答 + 状态页/终端 UI 演示`

如果现场麦克风噪声大，可允许操作员通过键盘触发固定问题，仍然能完整讲清“多模态联动”。

## 11. 对比赛叙事最重要的取舍

建议坚持以下取舍：

1. 不秀开放闲聊，专注防腐专业问题
2. 不秀过多模型切换，主打一条稳定主链
3. 不把评委注意力放在“算法名词”，而放在“现场巡检闭环”
4. 不追求高帧率观感，优先追求“识别可信 + 解释可信”
5. 不把 Windows 本地结果当正式结果，只用板端实测结果讲故事

## 13. 最终结论

**Phase 6 的最佳比赛形态，不是“再做一个新 UI”，而是把现有板端语音专家链路和两阶段视觉链路收束为一个统一的巡检终端。**

建议的最终版本是：

1. 评委看 `主画面`
2. 操作员看 `终端 UI`
3. 视觉主链用 `corrosion_two_stage_rt`
4. 问答主链用 `qwen2.5:0.5b + RAG`
5. 演示故事固定在“无网工业防腐巡检”
6. 全程以 Muse Pi Pro 板端实机为准

如果 Phase 6 按这个方案落地，比赛最终呈现出来的就不是“语音 Demo + 视觉 Demo”，而是一个真正像产品的 **全离线多模态防腐专家交互终端**。
