## 3. 对 `榨干.txt` 的第一轮修正

| 说法 | 当前结论 |
|---|---|
| “最大未挖掘点是原生流式合成” | **要降级表述**。当前仓库的 `StreamingCall()` 并不是真正双向流；`StartDuplexStream()` 仍未实现 |
| “临时 wav + 转码 + aplay 是最大性能税” | **被部分证伪**。它们有成本，但不是当前最大头 |
| “先改流式播放链就是最值钱” | **不完全对**。当前更大的痛点是初始化与 provider 现实约束 |

## 4. 已测出的链路成本

| 项目 | 结果 | 结论 |
|---|---|---|
| `save_to_file()` | `≈ 52ms` | 不是主瓶颈 |
| 当前 `22050Hz mono -> 48000Hz stereo` 转换 + 写新 wav | `≈ 29ms` | 有成本，但仍不是主瓶颈 |
| `tts_file_demo` 整体一次性执行（含初始化、warmup、合成、保存） | `≈ 30059ms` | C++ CLI 也绕不开初始化成本，不能把锅全甩给 Python |

## 5. 当前最重要结论

| 结论 | 含义 |
|---|---|
| `matcha_zh` 初始化极重 | 必须做“引擎常驻”，不能反复新建 |
| `threads=4` 明显不值 | 至少对当前 K1 + 短句，4 线程反而更差 |
| `provider=auto/spacemit` 当前不能作为正式主线 | K1 上仍被 EP 兼容问题阻断 |
| `matcha_zh_en` 的初始化和推理都更轻 | 它值得进入“候选正式路线”讨论 |
| 文件保存和转码不是第一优先级 | 先不要把主要精力误砍到小头上 |

## 6. 我建议的下一刀

| 优先级 | 动作 | 原因 |
|---|---|---|
| P0 | 设计“常驻单引擎 / 双引擎”策略 | 这是当前收益最大的方向 |
| P0 | 决定纯中文是否值得切到 `matcha_zh_en` 或作为竞赛极速模式 | 它的初始化与推理都更占优，但需补音质主观判断 |
| P1 | 再测 `matcha_zh_en` 在更多纯中文文本上的稳定性 | 防止只在单个样句上看起来更快 |
| P1 | 之后再测播放链 `aplay` 与更原生播放器路线 | 现在它不是第一瓶颈 |
| P2 | 若确有必要，再推进 C++ 热路径集成 | 当前证据还不足以说明“Python 是主瓶颈” |

## 7. 现阶段最稳的表达

**截至 2026-06-09 第一轮实测，`model-zoo-tts` 在 K1 上最该优先榨的不是“去掉临时 wav”，而是“接受 EP 当前不可用的现实，围绕 CPU 路线做引擎常驻、线程收敛与 preset 策略优化”。**
